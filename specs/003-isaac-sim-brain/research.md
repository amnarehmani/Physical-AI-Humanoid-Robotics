# Research: Module 3 (NVIDIA Isaac)

**Status**: Complete
**Date**: 2025-12-08

## 1. Technical Decisions

### Isaac Sim Environment
- **Decision**: Use **Omniverse** workflow with GUI-based scene setup first, then Python scripting.
- **Rationale**: 
    - Isaac Sim's GUI is intuitive for scene composition (lighting, physics).
    - Python API (`omni.isaac.core`) is powerful for automation but steeper learning curve.
    - Matches industry standard for synthetic data generation.
- **Assets**: Use built-in Nucleus assets (warehouse, humanoid USDs).

### Visual SLAM (VSLAM)
- **Decision**: Use **Isaac ROS Visual SLAM** (`isaac_ros_visual_slam`).
- **Rationale**: 
    - GPU-accelerated on Jetson/RTX.
    - Standard for NVIDIA-based robotics.
    - Replaces older CPU-based methods (ORB-SLAM) for performance.
- **Inputs**: Stereo Camera images (from Isaac Sim) + IMU.

### Navigation Stack
- **Decision**: Use **Nav2** with standard 2D Costmaps (for now).
- **Rationale**: 
    - "Walking" controllers are complex. We will abstract locomotion as a "floating base" or differential drive in simulation to focus on *planning* first.
    - Full bipedal walking requires reinforcement learning (Isaac Lab), which is out of scope for this specific module's "Navigation" focus. We assume a low-level controller handles balance.
- **Configuration**: Custom `nav2_params.yaml` with humanoid footprint.

## 2. Key Concepts to Cover

### Synthetic Data
- **Definition**: Data generated by simulation (Isaac Sim) to train AI models.
- **Advantage**: Perfect ground truth, infinite variety (domain randomization).

### Hardware Acceleration
- **Concept**: Offloading tasks (VSLAM, perception) to the GPU via CUDA/TensorRT.
- **Isaac ROS**: A collection of hardware-accelerated ROS 2 packages (GEMs).

### Humanoid Navigation
- **Challenge**: Non-circular footprint, complex kinematics.
- **Solution**: polygonal footprint in Nav2, conservative velocity limits.
